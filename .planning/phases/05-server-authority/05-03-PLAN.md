# Plan 05-03: Optimistic Client & Reconciliation

## Goal
Wire the client to use the server for authority while remaining responsive via optimistic prediction and delta-based reconciliation.

## Requirements
- **SRV-05**: Client applies optimistic visual updates before server responds (sprite position only).
- **SRV-06**: Client replaces full state with server snapshot on response (full-state-replace reconciliation).
- **SRV-07**: Irreversible side effects (death, messages, inventory changes) wait for server confirmation.

## Context
- **Input Gating**: To simplify reconciliation, input is disabled while a server request is "in-flight."
- **Snap-to-Truth**: Upon receiving the server delta, any client prediction is overwritten.
- **Predictive Combat**: Visual flashes and "hits" are shown immediately, but health changes and deaths are confirmed by the server response.

## Tasks
1. **Implement Input Gating**:
   - Update `src/game/input` and `src/game/systems/input-system.ts`.
   - Add a global `isRequestPending` flag.
   - Ignore new action inputs while `isRequestPending` is true.
2. **Implement Client Prediction**:
   - In the client turn loop, before sending the request:
     - Execute the player's action locally via `runActionPipeline` from `src/shared/pipeline.ts`.
     - Apply the resulting `StateDelta` to the client's `World` and `Grid`.
     - Trigger sprite movement animations optimistically.
3. **Handle Server Reconciliation**:
   - Upon receiving the `StateDelta` from the API:
     - Compare and apply the server's authoritative delta to the client world.
     - "Snap" any mismatched entities to the truth (e.g., if the server rejected a movement).
     - Process any effects that were *not* predicted (e.g., AI moves, player damage, items picked up).
4. **Deferred Irreversible Effects**:
   - Ensure `ENTITY_DIED` and message log events are only processed from the server-confirmed delta, not during prediction.
5. **E2E Testing**:
   - Create `tests/rendering/reconciliation.test.ts`.
   - Use mocked network requests (via `fetch-mock` or similar) to:
     - Verify prediction applies correctly.
     - Verify reconciliation corrects a desync (e.g., player predicted movement, but server rejected it).
     - Verify animations are triggered appropriately for both predicted and confirmed actions.

## Verification
- `npm run test tests/rendering/reconciliation.test.ts`: Reconciliation tests must pass.
- Manual verification: In the development environment, artificially introduce a server-side "wall" the client thinks is empty and confirm the player "snaps" back correctly.
