<objective>
Implement FOV computation using rot-js PreciseShadowcasting with three visibility states (visible/explored/hidden), distance-based brightness gradient, and entity visibility gating rules.

Purpose: FOV is central to roguelike gameplay — it determines what the player can see, creates tension through limited information, and drives tilemap alpha values. Entity visibility gating (enemies FOV-only, items FOV+explored) creates the asymmetry: static objects are remembered, mobile ones aren't.
Output: Pure FOV logic module ready to be consumed by the render system for tilemap tinting and sprite visibility.
</objective>

  <name>Task 1: FOV Computation & Explored Set Management</name>
  <files>src/rendering/fov.ts, tests/rendering/fov-visibility.test.ts</files>
  <behavior>
    - Open 5×5 room with player at center (2,2), radius 8 → all 25 tiles visible
    - Player at (2,2) with wall at (4,2) → tiles behind wall at (5,2), (6,2) NOT visible
    - Walls at FOV boundary ARE visible (player sees what blocks their view)
    - computeFovAlpha at player position → 1.0 (full brightness)
    - computeFovAlpha at FOV edge (distance=8) → ~0.5 (dimmed)
    - computeFovAlpha decreases monotonically with distance
    - Explored set accumulates: move 1 adds tiles, move 2 adds more, original tiles still present
    - clearExplored() resets the set to empty
  </behavior>
  <action>
Create `src/rendering/fov.ts`:

**computeFov(playerX: number, playerY: number, radius: number, lightPasses: (x: number, y: number) => boolean, exploredSet: Set&lt;string&gt;): Set&lt;string&gt;**
- Create new `PreciseShadowcasting(lightPasses)` instance
- Create empty `visibleSet = new Set<string>()`
- Call `fov.compute(playerX, playerY, radius, (x, y, r, visibility) => { ... })`
  - If `visibility > 0`: add `${x},${y}` to visibleSet AND to exploredSet
- Return visibleSet
- Note: walls at FOV boundary ARE included because PreciseShadowcasting reports them with visibility > 0

**computeFovAlpha(x: number, y: number, playerX: number, playerY: number, radius: number): number**
- Distance-based brightness gradient within FOV (per user decision)
- `dist = Math.sqrt((x - playerX) ** 2 + (y - playerY) ** 2)`
- `alpha = Math.max(0.5, 1.0 - (dist / radius) * 0.5)` — full brightness at center, ~0.5 at edge
- Returns alpha value for use in buildTilemap

**createExploredSet(): Set&lt;string&gt;** — returns new empty Set
**updateExplored(exploredSet: Set&lt;string&gt;, fovSet: Set&lt;string&gt;): void** — copies all fovSet entries into exploredSet
**clearExplored(exploredSet: Set&lt;string&gt;): void** — clears set (for dungeon regeneration, prevents pitfall of phantom explored tiles)

Tests use a mock `lightPasses` function (simple 2D boolean array — no PixiJS dependency):
- Open room: all tiles within radius are visible
- Wall blocking: tiles behind wall excluded from visible set
- Boundary wall: wall tile itself IS in visible set
- Alpha gradient: values decrease with distance, clamped at 0.5 minimum
- Explored accumulates across multiple computeFov calls
- clearExplored produces empty set

**isEntityVisible(pos: { x: number; y: number }, entityType: EntityVisibilityType, fovSet: Set&lt;string&gt;, exploredSet: Set&lt;string&gt;): { visible: boolean; alpha: number }**

Rules per user decision:
- `player`: always `{ visible: true, alpha: 1.0 }`
- `enemy`: visible ONLY in current FOV. In fovSet → `{ visible: true, alpha: 1.0 }`. Otherwise → `{ visible: false, alpha: 0 }`

Key asymmetry (from user decisions): "Items stay visible on explored tiles (dimmed like terrain) — helps player remember loot locations. Enemies are NOT shown on explored tiles — only visible when in current FOV."

Tests cover all combinations: each entity type × (in FOV, in explored only, in neither).
