<task type="auto" tdd="true">
  <name>Task 1: ECS types and EventBus implementation</name>
  <files>src/engine/ecs/types.ts, src/engine/events/types.ts, src/engine/events/event-bus.ts, src/engine/events/event-bus.test.ts</files>
  <behavior>
    - EventBus.on() registers a handler for an event type
    - EventBus.off() removes a specific handler
    - EventBus.emit() queues events (does NOT execute handlers immediately)
    - EventBus.flush() executes all queued event handlers in order
    - Recursive flush: if handlers emit new events during flush, those are processed in the same flush cycle
    - Max flush depth of 10 prevents infinite loops from circular event chains
    - Generic type map provides autocomplete for event types and handler signatures
    - defineComponent() creates a ComponentDef combining key string + Zod schema
  </behavior>
  <action>
    1. Create src/engine/ecs/types.ts:
       - `type EntityId = number`
       - `ComponentDef<T>` interface: `{ readonly key: string; readonly schema: ZodSchema<T> }`
       - `defineComponent(key, zodShape)` factory function that returns a typed ComponentDef — the ComponentDef IS the key passed to world.getComponent(entity, Position)
       - `type SystemFn = (world: World) => void` (use forward reference or import type)
       - `enum Phase { PRE_TURN = 'pre-turn', ACTION = 'action', POST_TURN = 'post-turn', RENDER = 'render' }`
       - Export `ComponentData<T>` as the inferred type from a ComponentDef's schema

    2. Create src/engine/events/types.ts:
       - `EngineEvents` interface with lifecycle events:
         - `ENTITY_CREATED: { entityId: EntityId }`
         - `ENTITY_DESTROYED: { entityId: EntityId }`
         - `COMPONENT_ADDED: { entityId: EntityId; componentKey: string }`
         - `COMPONENT_REMOVED: { entityId: EntityId; componentKey: string }`

    3. Create src/engine/events/event-bus.ts:
       - Generic `EventBus<TEventMap extends Record<string, unknown>>` class
       - Private `handlers: Map<string, Set<EventHandler>>` for subscriber registry
       - Private `queue: Array<{ type: string; event: unknown }>` for queued events
       - `on<K>(type, handler)` — adds handler to set for event type
       - `off<K>(type, handler)` — removes handler from set
       - `emit<K>(type, event)` — pushes to queue (no immediate execution)
       - `flush()` — drains queue, executes handlers, recurses if new events queued (max depth 10, throw on exceeded)
       - `clear()` — removes all handlers and clears queue (for testing/teardown)

    4. Create src/engine/events/event-bus.test.ts:
       - Test subscribe and receive: on() + emit() + flush() delivers event
       - Test queue semantics: emit() does NOT call handler until flush()
       - Test unsubscribe: off() prevents handler from being called on flush
       - Test multiple handlers: two handlers for same event type both fire
       - Test multiple event types: different types dispatch independently
       - Test recursive flush: handler that emits during flush triggers nested processing
       - Test max depth guard: circular event chain throws after 10 iterations
       - Test clear(): removes all handlers and pending events
  </action>
  <verify>
    <automated>npx vitest run src/engine/events/event-bus.test.ts --reporter=verbose</automated>
  </verify>
  <done>EventBus queues events on emit, flushes handlers in batch, supports typed generics, and all tests pass. defineComponent factory creates ComponentDef objects with Zod schema.</done>

<task type="auto" tdd="true">
  <name>Task 2: World class — entity lifecycle, component CRUD, queries, system registration</name>
  <files>src/engine/ecs/world.ts, src/engine/ecs/world.test.ts</files>
  <behavior>
    - world.createEntity() returns auto-incrementing EntityId and queues ENTITY_CREATED event
    - world.destroyEntity(id) removes entity and all its components, queues ENTITY_DESTROYED event
    - world.entityExists(id) returns true/false
    - world.addComponent(entity, def, data) stores component and queues COMPONENT_ADDED event
    - world.removeComponent(entity, def) removes component and queues COMPONENT_REMOVED event
    - world.getComponent(entity, def) returns typed component data or undefined
    - world.hasComponent(entity, def) returns boolean
    - world.query(...defs) returns EntityId[] of entities having ALL specified components
    - world.registerSystem(phase, system) adds system to phase's execution list
    - world.executeSystems(phase) runs all systems in registration order for that phase
    - Destroying an entity that doesn't exist is a no-op
    - Querying with no matching entities returns empty array
  </behavior>
  <action>
    1. Create src/engine/ecs/world.ts:
       - World class constructor accepts `EventBus<EngineEvents>`
       - Private `nextId: EntityId = 1`, `entities: Set<EntityId>()`
       - Private `stores: Map<string, Map<EntityId, unknown>>()` — Map-of-Maps storage per research Pattern 2
       - Private `systems: Map<Phase, SystemFn[]>()` — phase → ordered system list
       - createEntity(): increment nextId, add to entities set, emit ENTITY_CREATED, return id
       - destroyEntity(id): remove from entities, remove from ALL component stores, emit ENTITY_DESTROYED. No-op if entity doesn't exist.
       - entityExists(id): check entities set
       - addComponent(entity, def, data): get or create store for def.key, set data, emit COMPONENT_ADDED. Throw if entity doesn't exist.
       - removeComponent(entity, def): delete from store, emit COMPONENT_REMOVED. No-op if component not present.
       - getComponent(entity, def): look up store by def.key, return data cast to correct type, or undefined
       - hasComponent(entity, def): check existence in store
       - query(...defs): intersect entity ID sets from each def's store. Start with smallest store for efficiency. Return EntityId[].
       - registerSystem(phase, system): append to phase's system list
       - executeSystems(phase): iterate systems for phase in registration order, call each with `this`

    2. Create src/engine/ecs/world.test.ts:
       - **Entity lifecycle tests:**
         - createEntity returns sequential IDs (1, 2, 3)
         - entityExists returns true for created, false for uncreated
         - destroyEntity removes entity; entityExists returns false
         - destroyEntity on non-existent entity is no-op (no throw)
         - ENTITY_CREATED event is queued on create
         - ENTITY_DESTROYED event is queued on destroy
       - **Component CRUD tests:**
         - addComponent + getComponent round-trips data correctly
         - getComponent returns undefined for missing component
         - hasComponent returns true/false correctly
         - removeComponent removes the component; getComponent returns undefined
         - Mutating returned component object mutates the stored data (mutable references per CONTEXT.md)
         - addComponent throws if entity doesn't exist
         - COMPONENT_ADDED event is queued on add
         - COMPONENT_REMOVED event is queued on remove
       - **Query tests:**
         - query(Position) returns entities with Position component
         - query(Position, Health) returns only entities with BOTH
         - query with no matches returns empty array
         - query reflects entity destroy (destroyed entity no longer in results)
       - **System tests:**
         - registerSystem + executeSystems runs system function
         - Systems execute in registration order within a phase
         - Systems in different phases are independent
         - System receives World as argument
  </action>
  <verify>

  </verify>
