<task type="auto" tdd="true">
  <name>Task 1: Input manager with keyboard-to-action mapping</name>
  <files>src/game/input/types.ts, src/game/input/actions.ts, src/game/input/input-manager.ts, src/game/input/input-manager.test.ts</files>
  <behavior>
    - InputManager(bindings) creates manager with provided key→action map (defaults to DEFAULT_BINDINGS)
    - enable() attaches keydown listener to document
    - disable() removes keydown listener from document
    - enable() when already enabled is a no-op
    - disable() when already disabled is a no-op
    - Mapped keys trigger action callback via setActionHandler
    - Unmapped keys are ignored (no callback, no preventDefault)
    - Mapped keys call event.preventDefault() to suppress browser defaults (INP-03)
    - rebind(keyCode, action) updates the action map (INP-02)
    - unbind(keyCode) removes a binding
    - getBindingsForAction(action) returns all key codes bound to that action
    - Uses event.code for layout-independent key identification (not event.key)
    - WASD and arrow keys both map to movement by default
    - Period and Numpad5 map to WAIT
    - Escape maps to PAUSE
  </behavior>
  <action>
    1. Create src/game/input/types.ts:
       - `type ActionHandler = (action: GameAction) => void`

    2. Create src/game/input/actions.ts:
       - `enum GameAction { MOVE_NORTH, MOVE_SOUTH, MOVE_EAST, MOVE_WEST, WAIT, PAUSE }`
       - `DEFAULT_BINDINGS: Record<string, GameAction>` — ArrowUp/KeyW→MOVE_NORTH, ArrowDown/KeyS→MOVE_SOUTH, ArrowRight/KeyD→MOVE_EAST, ArrowLeft/KeyA→MOVE_WEST, Period/Numpad5→WAIT, Escape→PAUSE
       - `DIRECTIONS: Record<string, { dx: number; dy: number }>` mapping MOVE_NORTH→{0,-1}, MOVE_SOUTH→{0,1}, MOVE_EAST→{1,0}, MOVE_WEST→{-1,0}
       - `isMovementAction(action: GameAction): boolean` helper

    3. Create src/game/input/input-manager.ts:
       - `InputManager` class
       - Constructor: builds `Map<string, GameAction>` from bindings object
       - `setActionHandler(callback: ActionHandler)` stores callback
       - `enable()`: creates keydown handler, calls `document.addEventListener('keydown', handler)`, sets enabled flag
       - Keydown handler: look up `event.code` in action map, if found call `event.preventDefault()` (INP-03) then invoke action callback
       - `disable()`: calls `document.removeEventListener('keydown', handler)`, sets enabled flag false
       - `rebind(keyCode: string, action: GameAction)`: updates map entry
       - `unbind(keyCode: string)`: deletes map entry
       - `getBindingsForAction(action: GameAction): string[]`: filter map entries

    4. Write tests (RED then GREEN). Use `// @vitest-environment jsdom` directive:
       - Mapped key code dispatches correct GameAction via callback
       - Unmapped key code does not trigger callback
       - event.preventDefault() called for mapped keys (INP-03)
       - event.preventDefault() NOT called for unmapped keys
       - rebind() changes mapping — old binding gone, new binding active (INP-02)
       - unbind() removes a binding
       - getBindingsForAction() returns correct key codes
       - disable() stops dispatching (keydown after disable → no callback)
       - enable() after disable() resumes dispatching
       - Double enable() is safe (no duplicate listeners)
       - WASD and arrow keys both dispatch movement actions by default
  </action>

  </verify>

</task>

<task type="auto" tdd="true">
  <name>Task 2: Movement system with collision and bump-to-attack</name>
  <files>src/game/components/hostile.ts, src/game/components/blocks-movement.ts, src/game/systems/movement.ts, src/game/systems/movement.test.ts</files>
  <behavior>
    - Movement system processes a move intent (entityId, dx, dy) and resolves it against the grid
    - Cardinal movement: dx/dy values of -1, 0, or 1 in exactly one axis (MOV-01)
    - Target tile out of bounds → no movement
    - Target tile not walkable → no movement (MOV-02)
    - Target tile occupied by entity with Hostile component → emit BUMP_ATTACK event, no movement (MOV-03, per CONTEXT.md: automatic bump-attack, player stays in place)
    - Target tile occupied by entity with BlocksMovement component (non-hostile) → no movement, no attack
    - Target tile walkable and unoccupied (or occupied by non-blocking entity) → update Position component + grid.moveEntity
    - After movement, Position component and grid spatial index are in sync
    - BUMP_ATTACK event includes attacker and defender entity IDs
  </behavior>
  <action>
    1. Create src/game/components/hostile.ts:
       - `defineComponent('hostile', {})` — empty marker component (Zod z.object({}))
       - Entities with this component trigger bump-to-attack when another entity tries to move into their tile

    2. Create src/game/components/blocks-movement.ts:
       - `defineComponent('blocksMovement', {})` — empty marker component
       - Non-hostile entities that block movement (future: furniture, friendly NPCs)

    3. Create src/game/systems/movement.ts:
       - Export `createMovementSystem(world: World, grid: Grid, eventBus: EventBus): MovementSystem`
       - The returned object has a `processMove(entityId: EntityId, dx: number, dy: number): MoveResult` method:
         a. Get Position of entity from world
         b. Calculate target: targetX = pos.x + dx, targetY = pos.y + dy
         c. Check grid.inBounds(targetX, targetY) — if not, return 'blocked'
         d. Check grid.isWalkable(targetX, targetY) — if not, return 'blocked'
         e. Get entities at target via grid.getEntitiesAt(targetX, targetY)
         f. For each occupant (skip self):
            - If world.hasComponent(occupant, Hostile) → emit BUMP_ATTACK { attacker: entityId, defender: occupant } via eventBus, return 'bump-attack'
            - If world.hasComponent(occupant, BlocksMovement) → return 'blocked'
         g. Update Position: pos.x = targetX, pos.y = targetY
         h. Update grid: grid.moveEntity(entityId, oldX, oldY, targetX, targetY)
         i. Return 'moved'
       - Export `MoveResult = 'moved' | 'blocked' | 'bump-attack'`

    4. Write tests (RED then GREEN):
       - **Cardinal movement (MOV-01)**: entity at (5,5) + MOVE_NORTH → position becomes (5,4)
       - **All 4 directions**: verify dx/dy for N, S, E, W
       - **Walkability check (MOV-02)**: moving into unwalkable tile → 'blocked', position unchanged
       - **Out of bounds**: moving off grid edge → 'blocked', position unchanged
       - **Bump-to-attack (MOV-03)**: moving into tile with Hostile entity → 'bump-attack', BUMP_ATTACK event emitted, player position unchanged (stays in place per CONTEXT.md)
       - **BUMP_ATTACK event payload**: verify attacker and defender IDs
       - **Non-hostile blocking**: entity with BlocksMovement but no Hostile → 'blocked', no event
       - **Grid sync**: after successful move, grid.getEntitiesAt reflects new position
       - **Position sync**: after successful move, Position component matches new coordinates
       - **Empty target**: moving into empty walkable tile → 'moved'
       - **Non-blocking entity**: moving into tile with entity that has neither Hostile nor BlocksMovement → 'moved' (entity can coexist, e.g., items on ground)

    Tests should create a real Grid, mock World with minimal component data, and spy on eventBus.emit.
  </action>
